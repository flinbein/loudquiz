# ТЗ и план разработки: LoudQuiz

## Контекст

Игра для компании, где все физически в одной комнате. Игроки надевают наушники с музыкой, не слышат друг друга, капитан объясняет вопрос жестами/мимикой. Цель — создать веб-приложение с большим экраном (ведущий) и мобильными устройствами (игроки), связанными через VarHub WebSocket бэкенд.

## 2. Архитектура

### Роли
- **Host (ведущий)** — большой экран, управляет игрой, видит всё
- **Player (игрок)** — мобильное устройство, участвует в игре
- **Spectator (зритель)** — мобильное устройство, видит дублированный главный экран, не участвует в голосовании и ответах

### VarHub API (ws режим)

VarHub используется только для ретрансляции сообщений между браузерами. Вся логика — на фронтенде (у host). Host хранит авторитетное состояние игры и рассылает его всем участникам. Игроки отправляют только события (имя, ответ, нажатие кнопки). Зрители только получают состояние.

**Инициализация на host-стороне:**
```typescript
import { Varhub } from "@flinbein/varhub-web-client";

const hub = new Varhub("https://varhub.flinbein.ru/");
const room = hub.createRoomSocket({ integrity: "custom:loudquiz" });
await room.promise;
const roomId = room.id;  // передаётся игрокам через QR-код

// Соединение передаётся первым аргументом события — useConnection() не используем:
room.on("connection", (connection, ...params) => {
  // новое соединение создано (ещё не открыто)
  // можно вызвать connection.defer() для асинхронной проверки
});
room.on("connectionOpen", (connection) => {
  // соединение открыто — отправляем текущее состояние
  connection.send({ type: "syncState", state: publicGameState });
});
room.on("connectionClose", (connection, reason, wasOpen) => {
  handlePlayerDisconnect(connection);
});
room.on("connectionMessage", (connection, ...data) => {
  // data — аргументы, переданные через client.send(...)
  handlePlayerEvent(connection, data);
});

// Рассылка всем готовым соединениям:
room.broadcast({ type: "stateUpdate", state: publicGameState });

// Сообщение конкретному клиенту (например, капитану — текст вопроса):
captainConnection.send({ type: "captainSecret", questionText: "..." });

// Получить все соединения (опционально — с фильтром):
const allConnections = room.getConnections();
const readyConnections = room.getConnections(c => c.ready);
```

**На стороне игрока/зрителя:**
```typescript
const client = hub.join(roomId, { integrity: "custom:loudquiz" });
await client.promise;

// Получать сообщения от host:
client.on("message", (...data) => {
  handleStateUpdate(data);
});
// Отправить событие host (аргументы придут в connectionMessage как ...data):
client.send({ type: "submitAnswer", answer: "Мандарин" });

// Реконнект: создать новый hub.join с тем же roomId
// host узнаёт о реконнекте через connectionOpen и шлёт syncState
```

### Потоки данных

**Инициализация:**
```
[Player/Spectator] --join(roomId)--> [VarHub] --connectionOpen event--> [Host]
[Host] --connection.send(publicGameState)--> [new participant]
```

**Во время игры:**
```
[Player] --client.send(event)--> [Host]
  события: join(name, role), setTeam, ready, becomeCaptain,
           pickQuestion, submitAnswer, continue, noMoreIdeas

[Host] --room.broadcast(publicGameState)--> [All]   // после каждого изменения
[Host] --connection.send(captainSecret)-->  [только капитан]  // текст вопроса
```

**Таймеры и синхронизация часов:**

Устройства игроков могут иметь рассинхронизацию системных часов на несколько секунд, что заметно при отображении таймера. Решение — измерять смещение между часами клиента и host, и применять его к полученным timestamp.

**Шаг 1 — NTP-style ping-pong при подключении:**
```typescript
// Клиент отправляет:
client.send({ type: "ping", t1: Date.now() });

// Host отвечает немедленно:
room.on("connectionMessage", (conn, data) => {
  if (data.type === "ping") conn.send({ type: "pong", t1: data.t1, t2: Date.now() });
});

// Клиент получает ответ:
client.on("message", (data) => {
  if (data.type === "pong") {
    const t4 = Date.now();
    const rtt = t4 - data.t1;
    // clockOffset: сколько нужно прибавить к Date.now(), чтобы получить время host
    clockOffset = data.t2 - data.t1 - rtt / 2;
  }
});
```

Точность: `±rtt/2` ≈ ±10–50ms на локальном WiFi. Для таймеров 60–120 сек более чем достаточно.

**Шаг 2 — Уточнение offset из каждого broadcast:**

Host включает `serverNow` в каждый broadcast состояния. Клиент обновляет offset по скользящей средней (EWMA), не делая резких скачков:
```typescript
// В publicGameState добавляется:
interface PublicGameState {
  ...
  serverNow: number;  // Date.now() на host в момент отправки
}

// Клиент при получении каждого обновления:
const receivedAt = Date.now();
const rawOffset = data.serverNow - receivedAt;  // грубая оценка (без учёта задержки)
clockOffset = clockOffset * 0.9 + rawOffset * 0.1;  // EWMA сглаживание
```

**Использование:**
```typescript
function hostNow() { return Date.now() + clockOffset; }
function getRemaining(endsAt: number) { return Math.max(0, endsAt - hostNow()); }
```

Таймеры запускаются у Host. В `publicGameState` передаётся `timer: { endsAt: number }` (timestamp в часах host). Клиент вычисляет оставшееся время через `getRemaining(endsAt)`.
При реконнекте host шлёт актуальный gameState, клиент повторяет ping-pong для уточнения offset.

## 3. Роутинг

| URL | Описание |
|-----|----------|
| `/` | Главная — создать комнату или войти по коду |
| `/constructor` | Редактор вопросов и заданий (отдельный инструмент, независимый от игры) |
| `/{roomId}` | Экран игрока / зрителя |
| `/{roomId}/host` | Экран ведущего |

При переподключении игрок входит без повторного ввода имени (имя и роль сохраняются в sessionStorage по ключу roomId).

Если комната не существует или подключение не удалось — игрок видит ошибку и получает редирект на главную страницу `/`.

## 4. Схема состояния игры (GameState)

```typescript
type GamePhase =
  | "lobby"           // подключение игроков, выбор команды
  | "calibration"     // проверка наушников
  | "topic-suggest"   // предложение тем (только AI-режим)
  | "question-setup"  // ведущий готовит таблицу (только human host)
  | RoundPhase        // раунд стандартный
  | BlitzPhase        // раунд блиц
  | "finale"          // конец игры

type RoundPhase =
  | "round-captain"   // выбор капитана (без музыки)
  | "round-pick"      // таблица вопросов у всех игроков команды; капитан выбирает (без музыки)
  | "round-ready"     // наушники, музыка включается, все нажимают "Я готов" (включая капитана)
  | "round-active"    // таймер идёт, музыка играет
  | "round-answer"    // ввод финальных ответов (20-30 сек), музыка плавно затихает
  | "round-review"    // ведущий проверяет (только human host)
  | "round-result"    // итог раунда

type BlitzPhase =
  | "blitz-captain"   // выбор капитана + выбор порядка игроков (совмещённая фаза)
  | "blitz-pick"      // капитан выбирает задание (получает приватный список со словами)
  | "blitz-ready"     // наушники, музыка включается, все нажимают "Я готов"
  | "blitz-active"    // таймер идёт, музыка играет
  | "blitz-answer"    // ввод финального ответа (20 сек), музыка затихает
  | "blitz-result"    // результат блица
```

### Разделение GameState по ролям

Host хранит полный `FullGameState`. Каждому участнику рассылается только необходимая часть:

```typescript
// Рассылается всем (host, players, spectators):
interface PublicGameState {
  phase: GamePhase;
  activeTeamId: string;
  scores: Record<string, number>;               // teamId -> score
  players: PublicPlayerInfo[];                  // имя, команда, статус "ответил/нет"
  timer?: { endsAt: number };                   // timestamp конца таймера
  currentRound?: PublicRoundInfo;               // вопрос БЕЗ текста, только тема/сложность
  roundResult?: RoundResult;                    // результаты после раунда
  jokerUsed: Record<string, boolean>;           // teamId -> джокер израсходован навсегда
  jokerActivatedThisRound: Record<string, boolean>; // teamId -> джокер активирован в текущем раунде
}

// Рассылается только капитану текущего раунда (приватно):
interface CaptainPrivateInfo {
  questionText: string;
}

// Host видит полный объект (не рассылается):
interface FullGameState extends PublicGameState {
  questionTable: Question[][];          // все вопросы с текстами
  allAnswers: Record<string, string>;   // playerId -> ответ (во время раунда)
  settings: GameSettings;
}
```

Это гарантирует: игроки не видят текст вопроса (кроме капитана), не видят ответы других до финала раунда.

## 5. Настройки игры (GameSettings)

```typescript
interface GameSettings {
  backendUrl: string;           // по умолчанию https://varhub.flinbein.ru/
  teamCount: 1 | 2;            // одна команда или две
  gameMode: "standard" | "bonus" | "blitz";
  hostType: "human" | "ai";
  aiApiKey?: string;            // openrouter.ai key
  roundDuration: number;        // секунды (по умолчанию 60)
  answerDuration: number;       // секунды (по умолчанию 25)
  questionsPerTopic: number;    // кол-во вопросов в каждой теме (по умолчанию 4)
  topicCount: number;           // кол-во тем (по умолчанию 3)
  blitzRoundCount?: number;     // кол-во раундов блиц (по умолчанию 3)
  // длительность блиц-раунда не настраивается — вычисляется из размера команды
}
```

### Баланс вопросов при 2 командах

В режиме 2 команд каждая команда должна сыграть **одинаковое количество раундов**. Общее число вопросов в таблице: `topicCount × questionsPerTopic`. Если оно нечётное — последний вопрос в таблице помечается как "не используется" и недоступен для выбора. Таким образом обе команды сыграют ровно `(totalQuestions - 1) / 2` или `totalQuestions / 2` раундов каждая.

Пример: 3 темы × 4 вопроса = 12 → каждая команда по 6 раундов. 3 темы × 3 вопроса = 9 → игнорируется 1 → каждая по 4 раунда.

В режиме **одной команды** все вопросы доступны.

## 6. Формат JSON с вопросами

### Правила составления вопросов

- Вопрос должен иметь **несколько уникальных вариантов ответов** (не менее числа игроков в команде)
- Ведущий должен иметь возможность **проверить правильность ответа**
- Сложность определяется:
  - Насколько сложно передать смысл **жестами и мимикой**
  - Длинная формулировка → сложнее
  - Дополнительные **исключающие условия** (например: "Млекопитающее из трёх слогов", "Месяц с 31 днём")
  - **Каламбур или загадка** (например: "Город в стране восходящего солнца" вместо "Город в Японии")
- Диапазон сложности: 100 (легко) — 200 (очень сложно), кратно 10

```typescript
interface QuestionsFile {
  topics: Topic[];
  blitzTasks?: BlitzTask[];
}

interface Topic {
  name: string;
  questions: Question[];
}

interface Question {
  id: string;
  text: string;
  difficulty: number;       // от 100 до 200, кратно 10 — задаётся вручную в редакторе
  hint?: string;
  // hint: подсказка для ведущего-человека. Отображается на host-экране во время раунда
  // рядом с вопросом. Помогает ведущему понять, почему тот или иной ответ верный.
  // Например: "Принимаются: любые планеты Солнечной системы кроме Земли и Плутона"
  acceptedAnswers?: string[];
  // acceptedAnswers: используется только для AI-ведущего — список эталонных ответов,
  // которые ИИ использует при автоматической проверке как опорные примеры.
  // Без этого поля ИИ оценивает ответы только по тексту вопроса.
}

// BlitzTask — один раунд блица. Содержит набор вариантов слов/словосочетаний
// разной сложности. Капитан объясняет слова жестами, команда называет их вслух.
interface BlitzItem {
  text: string;       // слово или словосочетание — задание и ответ одновременно
  difficulty: number; // 200..400, кратно 10
}

interface BlitzTask {
  id: string;
  items: BlitzItem[]; // минимум 1; обычно 3–5 вариантов разной сложности
}
```

## 7. Экраны приложения

### 7.1 Главная страница (`/`)
- Кнопка "Создать игру" → форма настроек → создание комнаты VarHub → переход на `/{roomId}/host`
- Кнопка "Войти в игру" → поле ввода кода комнаты → переход на `/{roomId}`
- Кнопка "Редактор вопросов" → переход на `/constructor`
- Форма настроек при создании:
  - Режим игры (стандарт / бонус / блиц)
  - Количество команд (1 / 2)
  - Тип ведущего (человек / ИИ)
  - Если ИИ: поле для API Key
  - URL бэкенда — отображается как справочная информация, поле для изменения (сохраняется в localStorage)

### 7.2 Экран ведущего (`/{roomId}/host`)

**Фаза lobby** (включает выбор команды):
- QR-код на весь экран (ссылка на `/{roomId}`)
- Код комнаты текстом
- Список подключённых игроков, сгруппированный по командам (игроки выбирают сами)
- Кнопка "Начать игру" → при 2 командах первая ходящая определяется **случайно**

**Фаза calibration:**
- Инструкция "Игроки настраивают наушники"
- Список: кто подтвердил готовность
- Кнопка "Протестировать сигнал" — ведущий может воспроизвести ring.mp3 на своём экране для проверки
- Кнопка "Все готовы" (принудительный переход)

**Фаза topic-suggest** (только AI-режим):
- Прогресс-бар с таймером
- Поток предложений от игроков в реальном времени (имя игрока + предложенная тема)
- Когда таймер завершится (или все нажали "Нет идей") — ИИ автоматически выбирает N тем
- Список выбранных тем отображается на экране (с обоснованием)
- Затем автоматически запускается генерация вопросов (loading-экран), после чего переход к `round-captain`

**Фаза question-setup** (только human host):
- Загрузка JSON файла с вопросами
- Предпросмотр таблицы вопросов

**Фаза round-captain:**
- Кто активная команда
- Кто стал капитаном (первый нажавший)
- Предыдущие капитаны отмечены (нельзя 2 раза подряд)
- Без музыки

**Фаза round-pick:**
- Таблица вопросов у всех игроков активной команды (одинаковый вид); только капитан может кликнуть
- Если джокер не использован — капитан может активировать его здесь
- На главном экране: таблица + статус джокера активной команды
- Без музыки

**Фаза round-ready:**
- Музыка запускается у игроков
- Список: кто готов / не готов (все игроки команды, включая капитана)
- Когда все нажали "Я готов" — автоматически переходит к round-active; кнопка "Все готовы" на экране ведущего

**Фаза round-active:**
- Большой таймер (обратный отсчёт)
- Кто уже ответил / ещё нет
- Вопрос скрыт (виден только капитану на его устройстве)

**Фаза round-answer:**
- Прогресс-бар оставшегося времени (20-25 сек)
- Кто уже ответил финально

**Фаза round-review** (human host):
- Текст оригинального вопроса (вверху) + подсказка `hint` (если есть)
- Карточки с ответами игроков: имя, ответ, кнопки ✅ / ❌ / "объединить"
- Предварительный счёт раунда (обновляется при нажатии ✅/❌)
- Кнопка "Подтвердить"

**Правила оценки ответов (для ведущего):**
- ✅ Ответ **правильный** — ведущий должен иметь возможность проверить
- ❌ Ответ **расплывчатый** ("Африканский фрукт" на вопрос "Фрукт из Африки") — не засчитывается
- ❌ Ответ **одинаковый** с другим — ведущий объединяет ("Гагарин" и "Юрий Гагарин" = 1 ответ)
  - Исключение: вопросы на синонимы ("Смелый" → "Бесстрашный" и "Отважный" — оба считаются)
- ✅ Небольшие **опечатки** допустимы, если вопрос не требует точного написания
- Спорные моменты ведущий решает **на своё усмотрение** с объяснением

**Формула подсчёта очков стандартного раунда:**
```
score = difficulty × uniqueCorrectAnswers
```

**Фаза round-result** (перерыв между раундами):
- Текст оригинального вопроса (вверху)
- Карточки с ответами и статусами (✅/❌/объединены)
- Анимация начисления баллов
- **Текущий счёт обеих команд** — краткое подведение итогов раунда
- Если AI-ведущий: комментарий ИИ к раунду и обоснование спорных решений
- Кнопка "Следующий раунд" (любой игрок)

**Фаза finale:**
- Итоговые очки обеих команд
- Объявление победителя
- Статистика: кто сколько ответил, лучший игрок
- Кнопка "Перезапустить игру" — доступна **любому игроку**:
  - Если human host и вопросы были загружены из JSON — на host-экране появляется диалог с предложением загрузить новые вопросы (обязательно) или перемешать те же (опционально)
  - Если AI host — начинает новую игру с теми же настройками автоматически
  - Возвращает в фазу `lobby` с теми же настройками, игроки остаются подключёнными

### 7.3 Экран игрока (`/{roomId}`)

**Lobby** (выбор команды совмещён):
- Поле ввода имени (предзаполняется из localStorage `lastPlayerName`)
- Кнопка "Войти"
- После входа: список игроков по командам, кнопки "Присоединиться к команде X" / "Смотреть"
- Ожидание начала игры

**Calibration:**
- Иконка наушников (большая)
- Текст: "Наденьте наушники. Нажмите кнопку ниже, чтобы включить музыку и настроить громкость. Убедитесь, что не слышите собеседников."
- Кнопка "Включить / выключить музыку" (воспроизводит music.mp3)
- Кнопка "Проиграть сигнал" (воспроизводит ring.mp3 — для теста)
- Кнопка "Я готов"

**Topic-suggest** (только AI-режим):
- Прогресс-бар таймера вверху
- Поле ввода темы + кнопка "Предложить" (макс. 3 темы на игрока)
- Список уже предложенных своих тем
- Кнопка "Больше нет идей" — если все нажали, таймер завершается досрочно

**Round-captain:**
- Кнопка "Буду капитаном" (если не был капитаном в прошлом раунде)
- Отображение: кто уже нажал
- Без музыки

**Round-pick:**
- Таблица вопросов отображается у **всех игроков активной команды** (одинаковый вид)
- Только капитан видит кнопки выбора и кнопку "Активировать Джокер"; у остальных — таблица без интерактивности
- Уже взятые вопросы серые
- Без музыки

**Round-ready:**
- Иконка наушников, музыка включается автоматически
- Кнопка "Я готов" — у всех игроков команды, **включая капитана**
- Имя капитана выделено в списке готовности

**Round-active:**
- Для капитана: вопрос на экране (большой текст) + таймер; музыка играет
- Для остальных: таймер, иконка наушников, текстовое поле для ответа + кнопка "Ответить"

**Round-answer:**
- Музыка плавно затихает (~3 сек)
- Прогресс-бар таймера
- Текстовое поле + кнопка "Отправить ответ" — у всех игроков активной команды, **включая капитана**
- Если уже ответил: "Ответ отправлен, ждём остальных..."

**Round-review / round-result:**
- "Ведущий проверяет ответы..." / итог раунда
- Кнопка "Продолжить" (любой игрок)

## 8. Аудио-система

### Файлы
- `assets/music.mp3` — фоновая музыка во время раунда (зацикленная)
- `assets/ring.mp3` — сигнал начала/конца раунда

- Плавное затухание (~3 сек) при переходе в `round-answer`, `round-review`, `round-result`, `finale`
- Мгновенная остановка в `lobby`, `calibration`, `round-captain`, `round-pick` (музыка в этих фазах не играет)
- Калибровка: кнопка "Включить/выключить музыку" позволяет игроку вручную проверить наушники
- Звуковые сигналы (`ring.mp3`): воспроизводятся у **всех** (игроки, host, spectator): начало раунда, конец таймера, конец ввода ответов
- WakeLock API для предотвращения гашения экрана (активируется при входе в игру)

## 9. ИИ-ведущий

- API: openrouter.ai через библиотеку `@openrouter/sdk`
- Функции:
  1. **Выбор тем** — после topic-suggest phase, выбирает N тем из предложений игроков
  2. **Генерация вопросов** — по выбранным темам, с учётом правил составления вопросов
  3. **Проверка ответов + комментарий к раунду** — оценивает ответы игроков и даёт разбор
  4. **Генерация заданий для блиц** — слова/словосочетания для "Телефон блиц"
- API key вводится на главном экране, сохраняется в localStorage

### 9.1 Выбор тем

**Входные данные:**
```json
{
  "suggestions": [
    { "playerName": "Иван", "text": "Рок-музыка" },
    { "playerName": "Мария", "text": "Города России" }
  ],
  "topicCount": 3,
  "pastTopics": ["Кино", "Спорт"]
}
```

**Правила:**
- Выбирает ровно `topicCount` тем
- Может перефразировать ("Рок-музыка" → "Музыка"), объединять близкие предложения
- Игнорирует: слишком узкие темы, нецензурные, не подходящие для формата викторины
- Не использует темы из `pastTopics`
- Если предложений мало — дополняет универсальными темами (Наука, История, Природа и т.п.)
- Для каждой темы даёт краткое обоснование выбора (1 предложение)

**Формат ответа:**
```json
{
  "topics": [
    { "name": "Музыка", "reason": "Популярная тема, предложена несколькими игроками" },
    { "name": "Города России", "reason": "Предложена Марией, подходит для формата" },
    { "name": "Природа", "reason": "Добавлена для разнообразия" }
  ]
}
```

### 9.2 Генерация вопросов

**Входные данные:**
```json
{
  "topics": ["Музыка", "Города России", "Природа"],
  "questionsPerTopic": 4,
  "playersPerTeam": 4,
  "pastQuestions": ["Назовите музыкальный инструмент...", "..."]
}
```

**Правила составления вопросов** (из questions.md):
- Каждый вопрос должен иметь **не менее `playersPerTeam` уникальных правильных ответов**
- Ответы должны быть **проверяемыми** (ведущий или ИИ должны иметь возможность подтвердить)
- Сложность задаётся явно (100–200, кратно 10) и определяется:
  - Насколько сложно **передать смысл жестами и мимикой** (основной критерий)
  - Длинная формулировка → сложнее
  - **Исключающие условия** ("Млекопитающее из трёх слогов", "Месяц с 31 днём") → сложнее
  - **Каламбур или загадка** ("Город в стране восходящего солнца") → сложнее
- Вопросы не должны повторять `pastQuestions`
- Распределение сложности в теме: примерно равномерно от 100 до 200
- `hint` — подсказка для ведущего: что является допустимым/недопустимым ответом
- `acceptedAnswers` — 3–5 примеров правильных ответов (используются при автопроверке)

**Формат ответа:**
```json
{
  "topics": [
    {
      "name": "Музыка",
      "questions": [
        {
          "id": "q_music_1",
          "text": "Музыкальный инструмент, не состоящий из дерева",
          "difficulty": 160,
          "hint": "Принимаются: труба, флейта, саксофон, барабан. Не принимаются: гитара, скрипка (корпус — дерево), пианино (корпус — дерево)",
          "acceptedAnswers": ["Труба", "Флейта", "Саксофон", "Тромбон", "Барабан"]
        }
      ]
    }
  ]
}
```

### 9.3 Проверка ответов и комментарий к раунду

Выполняется единым вызовом ИИ после того, как все игроки прислали ответы. Заменяет ручную проверку ведущего-человека.

**Входные данные:**
```json
{
  "question": {
    "text": "Назовите человека, который летал в космос",
    "hint": "Принимаются реальные космонавты/астронавты",
    "acceptedAnswers": ["Гагарин", "Армстронг", "Терешкова"]
  },
  "answers": [
    { "playerId": "p1", "playerName": "Иван", "answer": "Гагарин" },
    { "playerId": "p2", "playerName": "Мария", "answer": "Юрий Гагарин" },
    { "playerId": "p3", "playerName": "Петр", "answer": "Я" },
    { "playerId": "p4", "playerName": "Анна", "answer": "Армстронг" }
  ]
}
```

**Правила проверки** (из rules.md):
- ✅ Ответ **правильный**: соответствует вопросу, поддаётся проверке
- ❌ Ответ **расплывчатый**: нельзя проверить, является ли он корректным ("Я", "Какой-то космонавт")
- ❌ Ответ **одинаковый**: семантически совпадает с другим → объединяются в группу, считается за 1 уникальный ответ
  - Исключение: если вопрос про синонимы/похожие варианты — оба принимаются
- ✅ Небольшие опечатки допустимы, если вопрос не требует точного написания
- Итоговый счёт: `difficulty × количество_уникальных_верных_групп`

**Формат ответа:**
```json
{
  "groups": [
    {
      "id": "g1",
      "accepted": true,
      "canonicalAnswer": "Гагарин",
      "playerIds": ["p1", "p2"],
      "note": "Ответы 'Гагарин' и 'Юрий Гагарин' объединены как одинаковые"
    },
    {
      "id": "g2",
      "accepted": false,
      "canonicalAnswer": "Я",
      "playerIds": ["p3"],
      "note": "Ответ невозможно проверить"
    },
    {
      "id": "g3",
      "accepted": true,
      "canonicalAnswer": "Армстронг",
      "playerIds": ["p4"],
      "note": null
    }
  ],
  "commentary": "Команда дала 2 уникальных верных ответа. Ответ 'Я' от Петра не принят — невозможно подтвердить. 'Гагарин' и 'Юрий Гагарин' объединены в один ответ."
}
```

Финальный счёт вычисляется на стороне host по формуле `difficulty × количество_принятых_групп`. ИИ счёт не считает.

### 9.4 Генерация заданий для блиц

**Входные данные:**
```json
{
  "count": 6,
  "difficultyRange": { "min": 200, "max": 400, "step": 10 },
  "pastTasks": ["Мандарин", "Велосипед"]
}
```

**Правила:**
- Каждое задание — набор слов/словосочетаний разной сложности, которые капитан объясняет **жестами и мимикой**
- Сложность определяется тем, насколько сложно изобразить слово без звука:
  - 200 — легко показать (фрукт, животное, простое действие)
  - 300 — умеренно (профессия, предмет быта, известная личность)
  - 400 — сложно (абстрактное понятие, малоизвестное слово, составное понятие)
- Абстрактные понятия ("Счастье", "Демократия") допустимы, но относятся к высокой сложности (≥ 350)
- `text` одновременно является заданием и правильным ответом; нормализованная форма для сравнения вычисляется автоматически (нижний регистр, `ё` → `е`)
- Не повторять слова из `pastTasks`
- Внутри каждого задания варианты должны охватывать разные уровни сложности

**Формат ответа:**
```json
{
  "tasks": [
    {
      "id": "b1",
      "items": [
        { "text": "Мороженое",       "difficulty": 200 },
        { "text": "Парикмахер",      "difficulty": 300 },
        { "text": "Цвет настроения", "difficulty": 400 }
      ]
    },
    {
      "id": "b2",
      "items": [
        { "text": "Яблоко",    "difficulty": 200 },
        { "text": "Водопад",   "difficulty": 250 },
        { "text": "Демократия","difficulty": 400 }
      ]
    }
  ]
}
```

## 10. Блиц-раунд ("Телефон блиц")

### Игровой флоу
1. Капитан выбирает (по кнопке "Буду капитаном")
2. Остальные выбирают порядок ("Буду следующим (N)")
3. Все надевают наушники и отворачиваются
4. Капитан выбирает задание → таймер стартует
5. У игроков появляется текстовое поле для досрочного ответа
6. Таймер истёк → 20 сек на финальный ответ + кнопка "Сдаюсь"
7. Автоматический подсчёт (без ведущего): точное совпадение без учёта регистра и ё/е, длина цепочки

### Таймер блиц-раунда

```
totalTime = (playersInTeam - 1) * 30  // секунды
```

Пример: команда из 5 человек (капитан + 4 игрока) → `(5 - 1) × 30 = 120 сек`.

### Подсчёт очков (автоматически)

**Цепочка верных ответов** — максимальная последовательность правильных ответов с начала (игрок 1, 2, … до первой ошибки). Ответы после ошибки в цепочку не входят.

Нормализация перед сравнением: привести к нижнему регистру + заменить `ё` на `е`.

```
// Если таймер закончился раньше, чем все ответили:
score = chainLength * taskCost

// Если все игроки успели ответить до конца таймера (досрочно):
score = chainLength * taskCost * (1 + timeLeft / totalTime)
// округление до целого числа
```

Пример: команда 5 человек (капитан ААА + 4 игрока), задание "Мандарин" (200), taймер 120 сек, все ответили досрочно, остаток ≈5 сек:
| Игрок | Номер | Ответ       | В цепочке |
|-------|-------|-------------|-----------|
| БББ   | 1     | Мандарин ✅  | да        |
| ВВВ   | 2     | мандарин ✅  | да        |
| ГГГ   | 3     | Апельсин ❌  | нет (цепочка оборвалась) |
| ДДД   | 4     | Мандарин ✅  | нет       |

chainLength = 2, все ответили досрочно → score = 2 × 200 × (1 + 5/120) ≈ **417**

## 11. Персистентность

Принцип: **sessionStorage** для данных, специфичных для текущей вкладки/сессии (позволяет одновременно открыть несколько вкладок с разными игроками для тестирования). **localStorage** только для подсказок и глобальных настроек.

| Данные | Хранилище | Ключ | Причина |
|--------|-----------|------|---------|
| Имя и роль игрока в комнате | **sessionStorage** | `player:{roomId}` | Изолировано по вкладке — разные вкладки = разные игроки |
| Последнее введённое имя | localStorage | `lastPlayerName` | Предзаполнение поля при входе в новую комнату |
| OPENROUTER_API_KEY | localStorage | `openrouterApiKey` | Глобальная настройка, не зависит от вкладки |
| URL бэкенда | localStorage | `backendUrl` | Глобальная настройка |
| Состояние редактора `/constructor` | localStorage | `constructorState` | Сохраняется между сессиями, есть кнопка сброса |
| Архив прошлых вопросов | загрузка JSON файла | — | Не сохраняется автоматически |
| Состояние игры | VarHub (host хранит, рассылает) | — | Авторитетный источник |

## 12. UX-детали

- **Адаптивность**: mobile-first (игроки) + wide-screen (host)
- **Реконнект игрока**: при потере связи автоматический переконнект, восстановление состояния через `syncState` от host
- **Реконнект host**: состояние игры хранится в памяти host-вкладки. Перезагрузка host-страницы во время игры приводит к потере состояния — предупредить пользователя. Для минимизации риска: сохранять `FullGameState` в `sessionStorage` на host-стороне при каждом изменении
- **WakeLock**: активируется при входе в игру
- **Headphone icon**: отображается всегда когда игрок должен быть в наушниках
- **Анимации**: начисление баллов, переходы между фазами
- **Цвета команд**: красная и синяя тема у карточек игроков

## 13. Дополнительные механики и фичи

### 13.1 Механика "Джокер" ✅

Каждая команда получает **1 карточку "Джокер"** на всю игру (в стандартном/бонусном режиме).

**Правила:**
- Капитан может активировать джокер **во время выбора вопроса** (в фазе `round-pick`)
- При активации `jokerActivatedThisRound[teamId] = true` — это видно всем в publicGameState
- В конце раунда (`round-result`): независимо от результата `jokerUsed[teamId] = true`, `jokerActivatedThisRound[teamId]` сбрасывается в `false`
  - Если команда набрала > 0 очков — очки **удваиваются**
  - Если команда набрала 0 очков — джокер сгорает без бонуса
- Использованный джокер (`jokerUsed = true`) больше не доступен

**Где отображается:**
- На host-экране: у каждой команды иконка джокера (доступен / активирован в раунде / использован)
- У капитана в round-pick: кнопка "Активировать Джокер" (если `jokerUsed = false`)
- В publicGameState: `jokerUsed: Record<teamId, boolean>` и `jokerActivatedThisRound: Record<teamId, boolean>`

### 13.2 Режим зрителя ✅

Третий тип входа на `/{roomId}` — "Смотреть". Зритель видит точную копию host-экрана, но без интерактивных элементов управления (нет кнопок "Засчитать", "Следующий раунд" и т.д.). Читает gameState напрямую, не влияет на игру.

### 13.3 Редактор вопросов `/constructor` ✅

Отдельная страница, независимая от игровой сессии. Состояние сохраняется в `localStorage` (`constructorState`).

**Ручное редактирование:**
- Добавление / удаление тем, вопросов, блиц-заданий
- Для каждого вопроса: текст, сложность (100–200, кратно 10), hint, acceptedAnswers
- Для блиц-задания: список вариантов (`items`), каждый вариант — текст + сложность (200–400, кратно 10); минимум 1 вариант
- Импорт JSON / Экспорт JSON / Кнопка "Очистить"

**AI-генерация (пошаговая, требует API Key):**

- Шаг 1 — Генерация тем: поле предложений + настройка N тем → ИИ возвращает список тем
- Шаг 2 — Генерация вопросов: на основе тем + `questionsPerTopic` → ИИ создаёт вопросы
- Шаг 3 — Генерация блиц-заданий: N заданий → ИИ генерирует наборы слов/словосочетаний разной сложности

После каждого шага результат доступен для ручной правки.

### 13.4 Статистика игроков в финале ✅

На экране финала:
- Таблица лидеров по команде
- Кто дал больше всего правильных ответов
- Кто чаще был капитаном
- Кто ответил первым в раунде (самый быстрый)

### 13.5 Backlog (нереализованные идеи)

- **"Кража"** — 0 очков у команды → противник может угадать вопрос за 15 сек и украсть очки
- **"Бонус за скорость"** — +30% если все ответили до половины таймера
- **Раунд "Дуэль"** — бонусный раунд 1-на-1 между командами
- **"Wild Card"** — скрытый вопрос со случайным множителем очков
- **Раунд "Сломанный телефон"** — из idea.md

Правила работы:
- Перед каждой фазой можно задавать уточняющие вопросы
- Переход к следующей фазе — только после подтверждения пользователя
- В процессе выполнения фазы — проверить поведение приложения в chrome, убедиться в соответствии по ТЗ.
- По возможности писать юнит-тесты; запускать в конце фазы
- При ошибке: найти причину → исправить → проверить тестами

**Ведение плана во время реализации:**
- По мере выполнения задач отмечать каждый выполненный пункт: `- [x] задача`
- После завершения фазы обновить её заголовок: добавить `✅` в начало
- Если в процессе обнаружены расхождения с планом — обновить план перед продолжением

### Фаза 5: Стандартный раунд (human host) ✅
**Цель:** Полный стандартный раунд от выбора капитана до результата.

**Задачи:**
- `round-captain`: кнопка "Буду капитаном", ограничение "не 2 раза подряд"; без музыки
- `round-pick`: таблица вопросов у всех игроков команды (одинаковый вид); только капитан может выбирать; джокер; без музыки
- `round-ready`: "Я готов" у всех игроков команды включая капитана; музыка стартует автоматически
- `round-active`: таймер (`endsAt` timestamp), вопрос у капитана, текстовое поле у всех игроков команды
- `round-answer`: прогресс-бар 20–25 сек, финальная отправка ответа у всех игроков включая капитана; музыка плавно затихает
- `round-review`: карточки ответов ✅/❌/"объединить", оригинальный вопрос вверху
- `round-result`: итог раунда + таблица счёта + "Следующий раунд"
- Баланс вопросов при 2 командах (нечётное кол-во → игнорировать последний)
- Юнит-тесты: `calculateScore()`, `balanceQuestions()`, логика таймера

**Результат для проверки:**
- Пройти 2 полных раунда двумя командами → ведущий проверяет ответы → счёт обновляется корректно
- Проверить джокер: активировать → очки удваиваются

### Фаза 6: Блиц-раунд ("Телефон блиц") ✅
**Цель:** Полный блиц-раунд с автоматическим подсчётом очков.

**Блиц-поток (реализовано):**
- `blitz-captain`: **совмещённая фаза** — сначала кто-то нажимает "Буду капитаном", затем остальные выбирают порядок (1..N кнопок); автопереход в `blitz-pick` когда капитан выбран И все остальные выбрали позицию
- `blitz-pick`: капитан получает **приватный** список заданий со словами (`blitzTaskList` — только ему, не рассылается остальным); выбирает задание и уровень сложности; остальные видят только стоимости
- `blitz-ready`: все надевают наушники, музыка включается, каждый нажимает "Я готов"; автопереход в `blitz-active`
- `blitz-active`: кольцевой сигнал при старте; капитан объясняет жестами (видит слово); игроки отвечают по очереди по blitzOrder; текстовое поле появляется только когда пришла очередь
- `blitz-answer`: кольцевой сигнал; все (кроме капитана) вводят финальный ответ + кнопка "Не знаю"; автопереход когда все ответили
- `blitz-result`: слово раскрывается, показаны ответы цепочкой, подсчёт очков

**Дополнительно реализовано:**
- `questionHistory`: накопительный массив в PublicGameState — какая команда ответила на какой вопрос и с каким счётом; используется для окраски ячеек QuestionTable
- Компонент `QuestionTable` (shared): ячейки окрашены по командам (красный/синий), активный вопрос подсвечен жёлтым, отвеченные показывают +score
- Компонент `TeamStatusBlock` (shared): список игроков с 👑 для капитана, номером блиц-порядка, статусом ответа (✓/✏️)
- Кольцевой сигнал: проигрывается на устройствах игроков и ведущего при переходе в round-active, round-answer, blitz-active, blitz-answer
- Капитан участвует в round-answer (тоже вводит ответ)

**Результат для проверки:**
- Пройти блиц-раунд с 4 игроками → проверить расчёт очков по всем случаям (досрочно / по таймеру)

### Фаза 7: Финал и перезапуск ✅
**Цель:** Финальный экран, статистика, перезапуск игры.

**Задачи:**

- Для human host с JSON: диалог загрузки новых вопросов
- Юнит-тесты: `computeGameStats()`, логика завершения

**Результат для проверки:**

### Фаза 8: ИИ-ведущий в игре ✅
**Цель:** Полная игра с AI host — автоматический выбор тем, проверка ответов, комментарии.

**Задачи:**
- `topic-suggest`: ИИ выбирает N тем после сбора предложений (вызов 9.1)
- Автогенерация вопросов: сразу после `topic-suggest` host вызывает ИИ (9.2), отображается loading-экран; после получения ответа переходим к `round-captain`

- `round-result` (AI): комментарий ИИ к раунду (разбор спорных ответов)
- Загрузка архива прошлых вопросов для исключения повторений
- Юнит-тесты: парсинг ответов на проверку, логика архива

**Результат для проверки:**
- Создать игру с AI host и API key из `.env`
- Пройти полную игру — темы выбираются автоматически, ответы проверяются ИИ

## 16. Верификация (dev-режим)

### Dev-режим (несколько вкладок)
- `npm run dev` запускает сервер, доступный по IP в локальной сети
- Открыть `http://localhost:5173/{roomId}/host` в одной вкладке
- Открыть `http://localhost:5173/{roomId}` в нескольких других вкладках — каждая работает как отдельный игрок (sessionStorage изолирован по вкладкам)
- Подключиться с телефона по `http://<local-ip>:5173/{roomId}` (QR-код ведёт на этот адрес)

**vite.config.ts** должен содержать:
```ts
server: { host: true }  // слушает 0.0.0.0
```

### Полный флоу
1. Создать комнату → `/{roomId}/host`
2. Открыть 4 вкладки-игрока → пройти lobby → calibration → раунд → результат
3. Проверить реконнект: закрыть и открыть вкладку игрока во время раунда
4. Проверить блиц-раунд с автоматическим подсчётом
5. Проверить финал → "Перезапустить игру"
6. Проверить ИИ-ведущего с API key из `.env`
7. Проверить зрителя: войти с ролью "Смотреть" — видит главный экран, не может отвечать
