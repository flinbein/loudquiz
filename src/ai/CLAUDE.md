# src/ai — AI-промпты и интеграция с OpenRouter

## Архитектура

Каждый AI-сценарий — набор из 4 файлов:

- `<name>.system.md` — системный промпт (роль, правила, стиль ответа)
- `<name>.user.md` — пользовательский промпт (входные данные)
- `<name>.ts` — сборка промпта, JSON-схема ответа, вызов `chatCompletionJSON`
- `<name>.test.ts` — тесты логики сборки и парсинга

Промпты импортируются через Vite `?raw`:

```ts
import systemPrompt from "./questionGeneration.system.md?raw";
```

Подстановка переменных — `renderTemplate(tmpl, vars)` из `template.ts`. Синтаксис: `{{placeholder}}`. Неизвестные плейсхолдеры остаются как есть (safe fallback).

Ответ всегда строго типизирован через `response_format: json_schema` с `strict: true`. Модель по умолчанию — `google/gemini-2.0-flash-001` (см. `client.ts`).

## Правила написания промптов

### system.md

- Начинай с роли: «You are a …» — коротко и конкретно.
- Затем одной строкой — задача.
- Правила — маркированным списком, императивом («must», «do not»).
- Язык ответа задавай плейсхолдером `{{language}}` в конце.
- Весь промпт — на английском. `{{language}}` управляет языком **ответа модели**.
- Не дублируй в system то, что есть в JSON-схеме (она уже принуждает структуру).

### user.md

- Только данные: списки, числа, контекст. Никаких инструкций.
- Списки форматируй через `formatList()` (`- item`). Пустые — как `(none)`.
- Плейсхолдеры — ровно те, что передаёт `.ts`-файл.

### .ts (сборка)

- Все числа приводи через `String(...)` — `renderTemplate` подставляет только строки.
- `SCHEMA` — с `additionalProperties: false` и полным `required` на каждом уровне (strict mode OpenRouter этого требует).
- Сигнатура функции: `(apiKey, input, language, model?)`.

## Рефакторинг промптов — рабочий процесс

1. **Запусти скилл `prompt-architect`** — он проанализирует текущий промпт и предложит фреймворк (CO-STAR, RISEN, RISE, TIDD-EC, RTF, CoT, CoD) под интент (create / transform / reason / critique / clarify / agentic).
2. **Либо `prompt-master`** — если нужно адаптировать под конкретную модель (Gemini 2.0 Flash ≠ Claude ≠ GPT). Извлекает 9 измерений интента, маршрутизирует под target-инструмент.
3. Применяй правку только к одному из `system.md` / `user.md` за раз. Не перемешивай роли.
4. После рефакторинга — обнови/запусти `<name>.test.ts`. Моки в тестах проверяют именно содержимое собранного промпта.
5. Если меняется структура ответа — синхронно правь `SCHEMA` в `.ts` и типы в `@/types/ai`.

### Что не трогать при рефакторинге

- `{{language}}` — игровая логика зависит от языка ответа модели.
- Числовые ограничения (`difficulty 100–200 step 10`, `playersPerTeam` и т.п.) — это контракт с UI.
- Формат списков (`- item`) — тесты строго на него завязаны.

## Известные особенности моделей

- **Gemini 2.0 Flash** (по умолчанию): хорошо держит JSON strict, но иногда игнорирует тонкие стилистические инструкции. Формулируй правила прямо и коротко.
- Прежде чем добавлять новый `<name>.system.md`, посмотри существующие — стиль и тон должны совпадать (игра party-quiz, не энциклопедия).

## Тестирование

- `*.test.ts` проверяют сборку промпта (подстановки, форматирование списков) и парсинг ответа — не сетевые вызовы. `chatCompletionJSON` мокается.
- E2E с реальным OpenRouter не запускается в CI.
